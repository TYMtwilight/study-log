# 楽観的ロック

データを更新する際、事前にロックをかけずに処理を進め、書き込む直前に衝突を検知する方式。

---

## 仕組み

1. データを取得するときに `VERSION` の値を手元に記憶する
2. ロックをかけずに処理を進める
3. 書き込む直前に「取得時の VERSION」と「現在の VERSION」を比較する
4. 一致していれば更新し、VERSION を +1 する
5. 一致していなければ「他の処理が先に書き換えた」と判断してエラーにする

```
取得時:  VERSION = 1  ← この値を手元に記憶しておく
              ↓
         （処理中...）
              ↓
書込時:  WHERE VERSION = 1  ← 手元の値と一致するか確認
              ↓
    一致した → UPDATE + VERSION を 2 に上げる → 成功
    一致しない → 誰かが先に書き換えた → エラー
```

---

## 悲観的ロックとの違い

|  | 悲観的ロック | 楽観的ロック |
|---|---|---|
| 考え方 | 衝突は必ず起きると考える | 衝突はめったに起きないと考える |
| 方式 | 取得時にロックをかけてブロックする | ロックをかけず書込時に衝突を検知する |
| 向いている場面 | 衝突が頻繁に起きる場合 | 衝突がほとんど起きない場合 |

---

## Spring Batch での用途

`BATCH_*` テーブルの各 `VERSION` カラムがこの仕組みの実装。複数スレッドや複数サーバーが同じジョブを同時に操作してしまった場合の安全装置として機能する。

アプリ独自のテーブル（`study_logs` や `daily_summaries` など）に `VERSION` カラムが必要かどうかは、そのテーブルを複数の処理が同時に書き換える可能性があるかどうかで判断する。
